# Shared Development Guide

This document defines core development standards that apply across all AI tools working on this codebase.

## Git Discipline

### Commit Standards

- **Never use `git add -A` or `git add .`** - Always stage specific files
- **Commit message format:** `JIRA-TICKET: Brief description`
  - Example: `CIVIMM-123: Add payment retry logic for failed transactions`
- **No AI attribution in commits** - Do not include "Generated by AI" or similar
- **One logical change per commit** - Keep commits focused and atomic

### Branch Naming

- Feature branches: `JIRA-TICKET-brief-description`
- Example: `CIVIMM-246-gocardless-integration`

### Before Pushing

1. Run tests: `phpunit5`
2. Run linter: `./bin/phpcs.phar --standard=phpcs-ruleset.xml <changed-files>`
3. Review your own diff: `git diff --staged`

## Code Quality Standards

### Testing Requirements

- **All new features must have tests**
- **All bug fixes must have tests** that reproduce the bug
- Tests must pass before merging
- Use the existing `BaseHeadlessTest` base class

### PHPStan Compatibility

- Code must pass PHPStan analysis
- Use `@phpstan-param` and `@phpstan-return` annotations when needed for dynamic types
- See `.ai/civicrm.md` for CiviCRM-specific PHPStan patterns

### Linting

- Code must pass PHPCS with Drupal/CiviCRM standards
- Run: `./bin/phpcs.phar --standard=phpcs-ruleset.xml <file>`
- Auto-fix: `./bin/phpcbf.phar --standard=phpcs-ruleset.xml <file>`

## Security Guidelines

### SQL Injection Prevention

- **Never** concatenate user input into SQL queries
- Use CiviCRM API4 which handles escaping automatically
- When raw SQL is necessary, use parameterized queries

### XSS Prevention

- Escape all output in templates
- Use CiviCRM's built-in escaping functions
- Never trust user input in HTML contexts

### Sensitive Data

- Never commit credentials, API keys, or secrets
- Never log sensitive payment information
- Follow PCI compliance guidelines for payment data

## Pre-Merge Validation Checklist

Before requesting merge:

- [ ] All tests pass (`phpunit5`)
- [ ] No linting errors (`./bin/phpcs.phar --standard=phpcs-ruleset.xml`)
- [ ] PHPStan passes (if configured)
- [ ] Code has been self-reviewed
- [ ] Commit messages follow format
- [ ] No debug code left in (var_dump, console.log, etc.)
- [ ] No commented-out code blocks
- [ ] Documentation updated if needed

## Review Feedback Handling

When receiving review feedback:

1. **Think critically** - Don't blindly accept or reject suggestions
2. **Ask questions** - If feedback is unclear, ask for clarification
3. **Explain reasoning** - If you disagree, explain why with evidence
4. **Address all comments** - Don't leave review comments unresolved

## Code Style

### PHP

- Follow PSR-12 with Drupal/CiviCRM modifications
- Use meaningful variable and function names
- Keep functions focused and small
- Document complex logic with comments

### Class Organization

- `CRM_*` classes: PSR-0 in root directory
- `Civi\*` classes: PSR-4 in `Civi/` directory
- See `.ai/civicrm.md` for CiviCRM patterns
